# План интеграции реальных конференц-систем (DCS100 / DCS150)

## 1) Что видно по текущему коду и временным файлам

- В текущем MVP сервер хранит только «внутреннее» состояние микрофонов и переключает его локально (`toggle`) без связи с реальным оборудованием.
- Для внешних интеграций уже есть TCP line-протокол (`SET MIC <micId> TOGGLE`) на порту `31415`, но это тоже логический toggle, а не драйвер железа.
- Во временных директориях лежат два разных драйвера:
  - `DCS100-temp/RMS_DCS100.js`: **TCP**, бинарные кадры с `STX/ETX`, очередь отправки, watchdog, парсинг уведомлений о состоянии микрофонов.
  - `DCS150-temp/RMS_DCS150.js`: **UDP**, bind локального интерфейса, keepalive/sync, уведомления `FUNC_NOTIFY (0x87)` по состоянию микрофона.

Вывод: протоколы уже существенно отличаются по транспорту, фреймингу и жизненному циклу соединения; значит нужна расширяемая адаптерная архитектура (не if/else на 2 типа).

## 2) Целевая архитектура (масштабируемая на N типов систем)

### 2.1. Слой `ConferenceDriver` (единый контракт)

Добавить абстракцию драйвера в `server/conference/`:

```ts
interface ConferenceDriver {
  readonly type: string;
  readonly capabilities: DriverCapabilities;

  start(config: DriverConfig): Promise<void>;
  stop(): Promise<void>;

  setMicState(micId: number, state: 'ON' | 'OFF'): Promise<void>;
  setCentralParam?(key: string, value: unknown): Promise<void>;
  setMicParam?(micId: number, key: string, value: unknown): Promise<void>;
  toggleMic?(micId: number): Promise<void>; // optional

  // events
  on(event: 'micState', cb: (e: { micId: number; state: 'ON' | 'OFF'; raw?: unknown }) => void): void;
  on(event: 'health', cb: (e: { status: 'connected' | 'disconnected' | 'degraded'; reason?: string }) => void): void;
  on(event: 'error', cb: (e: { message: string; raw?: unknown }) => void): void;
}
```

`DriverCapabilities` (минимум):

```ts
{
  micOnOff: true,            // обязательно для всех
  micStateFeedback?: boolean,

  // центральный блок
  allMicsOff?: boolean,
  centralVolume?: boolean,
  speechMode?: boolean,
  maxOpenMics?: boolean,
  centralEq?: boolean,

  // микрофонный пульт
  micSensitivity?: boolean,
  micTimer?: boolean,
  micEq?: boolean,
  micVip?: boolean
}
```

### 2.2. `ConferenceManager` (оркестратор)

Один менеджер в сервере:

- читает настройки активной системы;
- создает **ровно один** активный драйвер;
- пробрасывает команды из API/UI в драйвер;
- принимает события от драйвера и обновляет runtime-state + лог + TCP broadcast;
- умеет hot-switch: остановить старый драйвер, поднять новый.

### 2.3. Реестр драйверов

Чтобы легко добавлять новые системы:

```js
const registry = {
  dcs100: createDcs100Driver,
  dcs150: createDcs150Driver,
  // future: boschDicentis, taidenX, ...
};
```

Добавление нового типа = новый модуль + регистрация в реестре, без изменений основной логики API.

## 3) Настройки: одновременно только одна активная конференц-система

Добавить в конфиг проекта секцию, например:

```json
"conference": {
  "enabled": true,
  "type": "dcs100",
  "deviceIp": "192.168.1.50",
  "bindIp": "192.168.1.10", 
  "options": {
    "debug": false,
    "timeoutMs": 1500
  }
}
```

Где:

- `type` и `deviceIp` — обязательные (по условию задачи).
- `bindIp` — опционален, но нужен для UDP-систем вроде DCS150.
- `options` — vendor-specific параметры.

API:

- `GET /api/conference/settings`
- `POST /api/conference/settings`
- `GET /api/conference/status` (active driver, health, capabilities)

При сохранении настроек:

1. валидация (`type` существует в реестре, IP валидный);
2. пробный старт драйвера;
3. если старт успешен — делаем активным и сохраняем;
4. если нет — возвращаем ошибку и оставляем предыдущую рабочую конфигурацию.

## 4) Как связать существующую модель микрофонов с реальным железом

### 4.1. Нормализация `micId`

`micId` должен быть **только числовым** (адрес пульта в системе) и использоваться одновременно:

- в UI как редактируемый идентификатор,
- в API,
- в драйвере для команд к железу.

Решение в структуре микрофона:

```json
{
  "id": "...",
  "micId": 12,
  "micText": "12",
  "label": "Стол президиума"
}
```

Валидация на сервере: `micId` — целое число в допустимом диапазоне конкретной системы (например 1..128 для DCS100, если подтвердится протоколом).

### 4.2. Командная модель

Вместо «toggle всегда» сделать базовую команду:

- `POST /api/microphones/:id/on`
- `POST /api/microphones/:id/off`
- `POST /api/microphones/:id/toggle` (фолбэк/удобство)

Где `:id` — это именно числовой `micId` пульта.

Почему это важно:

- toggle неидемпотентен;
- при потере обратной связи проще гарантировать конечное состояние через `setMicState`.

### 4.3. Источник истины состояния

Состояние в UI должно быть приоритетно от драйвера (если драйвер дает feedback):

1. отправили команду — пометили `pending`;
2. пришло подтверждение/notify — фиксируем `ON/OFF`;
3. timeout — `unknown` + предупреждение в UI/логе.

Для систем без feedback можно оставить optimistic update, но явно маркировать такой режим.

## 5) Реализация под DCS100 и DCS150

## 5.1. DCS100 адаптер

Использовать код из `RMS_DCS100.js` как основу и вынести в драйвер:

- TCP connect на `8088`;
- очередь отправки + watchdog;
- `micOn/micOff` через функции `0x31/0x32`;
- парсинг `0x21` (битовые маски состояния);
- эмит `micState` при обнаружении изменений.

Что обязательно до прод-использования:

- унифицировать reconnection/backoff;
- добавить строгую проверку CRC и длины;
- ограничить размер очереди и policy (drop oldest / reject).

## 5.2. DCS150 адаптер

Использовать `RMS_DCS150.js`:

- UDP TX/RX (`18092/18093`), bind локального IP;
- startup `sync()` с локальным IP в payload;
- keepalive ответы `0x98`;
- состояние микрофонов по `0x87`;
- команды ON/OFF `0x01/0x02`.

Нужно добавить:

- валидацию `bindIp` (чтобы реально существовал локальный интерфейс);
- защиту от packet storm (rate limit логов);
- health-статус при отсутствии keepalive/notify длительное время.

## 6) Опциональные возможности (capabilities)

Так как функциональность железа разная, UI/API должны работать через capabilities:

- Базовый минимум (всегда): `micOnOff`.

### 6.1. Настройки центрального блока (отдельное окно)

Да, имеет смысл выделить отдельное окно/модалку **«Настройки центрального блока»**:

- вкладка «Основные»: режим работы, максимум открытых микрофонов;
- вкладка «Аудио»: общая громкость, EQ и т.п.;
- вкладка «Служебные»: sync/reboot и др. (если поддерживается).

Состав полей строится динамически по `capabilities` + `schema` от драйвера.

Пример ответа для UI:

```json
{
  "central": {
    "tabs": [
      {
        "id": "main",
        "title": "Основные",
        "fields": [
          {"key": "speechMode", "type": "select", "options": ["FIFO", "NORMAL", "FREE"]},
          {"key": "maxOpenMics", "type": "number", "min": 1, "max": 6}
        ]
      }
    ]
  }
}
```

API для центрального блока:

- `GET /api/conference/central/schema`
- `GET /api/conference/central/settings`
- `POST /api/conference/central/settings` (частичное обновление)

### 6.2. Настройки микрофонного пульта (отдельное окно)

Также да, отдельно нужно окно **«Настройки пульта»** для выбранного `micId`:

- вкладка «Аудио»: чувствительность, локальная громкость, EQ;
- вкладка «Логика»: таймер, приоритет/VIP, автoвыключение и т.п.

Поля тоже строятся динамически по возможностям конкретного драйвера.

API для пульта:

- `GET /api/microphones/:id/schema`
- `GET /api/microphones/:id/settings`
- `POST /api/microphones/:id/settings`

Если параметр системой не поддерживается — UI его не показывает.

Это позволит подключать новые системы без «ломающих» изменений UI.

## 7) Предложение по поэтапному внедрению

1. **Этап A (рефакторинг ядра):**
   - добавить `ConferenceManager`, `ConferenceDriver` интерфейс, реестр драйверов;
   - вынести текущий toggle-flow в менеджер.
2. **Этап B (базовый hardware control):**
   - реализовать `dcs100` и `dcs150` драйверы с командами on/off;
   - добавить настройки `type + deviceIp (+ bindIp)` и health endpoint.
3. **Этап C (настройки центрального блока и пультов):**
   - добавить schema-driven API для central/mic settings;
   - добавить отдельные UI-окна с вкладками и динамическими полями.
4. **Этап D (feedback и надежность):**
   - связать notify от драйверов с runtime-state;
   - добавить `pending/unknown`, timeout policy, расширенные логи.
5. **Этап E (расширяемость):**
   - capability-driven UI,
   - шаблон/гайд для добавления новых драйверов,
   - контрактные тесты для драйверов.

## 8) Минимальный набор тестов перед вводом в эксплуатацию

- Юнит-тесты драйверов:
  - корректная сериализация кадров;
  - обработка некорректных пакетов;
  - reconnection и очередь.
- Интеграционные тесты менеджера:
  - переключение активного `type` на лету;
  - отказ при неверном IP/недоступности;
  - корректное обновление runtime-state.
- E2E smoke:
  - из UI включить/выключить микрофон;
  - получить событие в TCP integration;
  - увидеть актуальное состояние после notify от железа.

---

## 9) Ключевые риски и как их закрыть

- **Toggle-дрейф состояния** между UI и железом → перейти на `set ON/OFF` + feedback-first.
- **Разные транспорты/тайминги** → инкапсулировать в драйверах, а не в API-слое.
- **Рост числа вендоров** → registry + capability flags + контракт драйвера.
- **Сетевая нестабильность** → health-мониторинг, backoff, bounded queue, observability.

Итог: предлагаемая схема поддержит одновременно только одну активную конференц-систему (тип + IP из настроек), но останется расширяемой для подключения новых систем с отличающимися протоколами и опциональными функциями.
